# デートBot 要件定義書（1枚）

| 項目 | 内容 |
|------|------|
| 作成日 | 2026-07-17 |
| バージョン | 1.2（店舗CSV廃止。エリア＋好み要約のみ） |
| 期限 | 今週末（Phase A 完成） |

---

## 1. 目的・成功条件

**目的**  
自分と彼女向けに、東京23区のデートプラン提案・お店／場所選びを支援する対話Botを作る。

**成功条件（今週末）**
- Colab上で Fine-tuning 〜 複数ターン対話テストまで動く（Phase A）
- 彼女に見せられる品質感がある
- 公開GitHubリポジトリにコード／説明を載せる（UIは後続でも、リポジトリ公開は目標に含める）

---

## 2. 利用者・使い方

| 項目 | 決定内容 |
|------|----------|
| 利用者 | 自分と彼女（呼称: 彼氏＝「リョウスケ君」、彼女＝「こゆたん」） |
| 入力 | 複数ターン対話（予算・時間帯・シチュエーションは都度ヒアリング） |
| 出力 | **案を3つ**、それぞれに**理由つき** |
| 口調 | フランク |
| 失敗時 | 「もっと条件を教えて」と聞き返す、またはエラー表示 |

---

## 3. 対象範囲

| やる | やらない |
|------|----------|
| デートプラン提案 | 予約代行 |
| お店・場所選び（エリア＋好みに基づき提案。店リストは持たない） | 実在店の在庫／空席のリアルタイム確認 |
| 東京23区 | 恋愛相談の深掘り |
| なるべく安価（学生でも行ける）を最優先制約 | 個人情報の一切の取り扱い |

**可変条件（都度聞く）**: 予算、時間帯、シチュエーション  
**固定寄り制約**: 安価志向、東京23区、個人情報ゼロ（絶対条件）

---

## 4. データ・モデル

| 項目 | 決定内容 |
|------|----------|
| 学習データ元 | 二人の好みメモ、過去のLINE（**生ログ全文は入れない**） |
| 学習に入れる形 | 「好み・禁止事項・定番デート」に**要約した文だけ** |
| 初期件数 | まず **10件**（input/output 形式） |
| モデル分担 | **B案**: 自前モデル＝口調・好み／**Gemini無料枠＝プラン具体化・店名案** |
| 店舗情報 | **使わない**。エリア（渋谷・上野・新宿／お台場〜品川）と好み要約だけで提案。実在店名は Gemini が一般知識から出す（不確実なら「候補」と明示） |
| 品質感 | お遊び可だがほどほど |

### 匿名化・個人情報ルール（絶対）

| ルール | 内容 |
|--------|------|
| 個人情報 | **一切扱わない**（電話・メール・住所・自宅寄りの駅名など削除） |
| 二人の呼称 | 彼氏 →「リョウスケ君」、彼女 →「こゆたん」 |
| 友人・家族名 | 削除、または「友人A」等に置換 |
| 勤務先・研究室名 | **出さない** |
| 学習データ | 要約のみ。LINE生ログ全文はリポジトリにも学習セットにも入れない |

## 7. GitHub公開ポリシー（企業・民間の慣行に合わせる）

一般的な会社・スタートアップの公開リポジトリでは次が標準に近い。

| 種別 | 方針 | 本プロジェクト |
|------|------|----------------|
| ソースコード・ノート・README | 公開してよい | **載せる** |
| APIキー・トークン・`.env` | **絶対に載せない**（秘密情報管理が基本） | **載せない**（`.gitignore` + 環境変数／Colabシークレット） |
| 顧客・ユーザーの生データ、チャット原文 | **載せない** | LINE生ログ・個人メモ原文は**載せない** |
| 学習用データ | ダミー／匿名化サンプルのみ、または非公開 | **匿名化サンプルCSVのみ**（再現用の最小例）。本番用要約データはローカル／Colabのみ可 |
| 学習済みモデル本体 | リポジトリに直置きしないことが多い（HF Hubや配布手順で分離） | **載せない**（取得・学習手順をREADMEに書く） |
| 社名・内部URL・未公開仕様 | 載せない | 勤務先・研究室名は出さない（上記ルールと一致） |

**本プロジェクトの確定方針（企業寄せ）**  
→ **コード + README + 匿名化サンプルCSV（ダミーまたはごく一部）まで公開。秘密情報・生ログ・本番データ・モデル本体は載せない。**

---

## 8. 受け入れ基準（Phase A）

1. 複数ターンで条件（予算・時間帯等）を聞ける  
2. 条件を踏まえ、案3つ＋理由を出せる  
3. 「安価・学生向け・東京23区」が提案に反映される気配がある  
4. 予約代行・在庫確認・深い恋愛相談には乗らない（拒否または回避）  
5. 個人情報を含まない学習データで動作する  
6. 自前モデル（口調・好み）と Gemini（具体プラン・店）の分業が動く  
7. 追加学習の手順がREADMEまたはノートに残っている  
8. 公開リポジトリに秘密情報・生ログが入っていない
